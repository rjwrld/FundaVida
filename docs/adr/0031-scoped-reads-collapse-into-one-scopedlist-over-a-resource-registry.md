# Scoped reads collapse into one scopedList over a RESOURCE_READ registry

_Proposed (architecture review 2026-07-03). Deepens ADR-0008; does not change the token contract._

Every entity read in `src/data/api/*.ts` was the same four-line pipeline — `await delay()`, read the store slice, take the scope token from `scopeFor(role)[resource]`, `applyScope(resource, token, items)`, `applyFilters` — copied across ten `list()` methods and four `get()` methods, differing only in the store slice, the resource string, the item type, and the entity-local `applyFilters` predicate. The resource was an unchecked magic string index into `scopeFor(role)`. The pipeline was never a module: it was a shape ten files re-expressed. Deleting any one file's pipeline concentrated nothing; it just relocated four identical lines.

The pipeline becomes two bare helpers, `scopedList(resource, filters, applyFilters?, scopeOverride?)` and `scopedGet(resource, id, scopeOverride?)`, in `src/data/api/scopedRead.ts`. Each entity's `list`/`get` is a one-line delegation; the entity file keeps only its `Filters` interface and `applyFilters` predicate — entity-local knowledge stays entity-local, and no mega-file is created. Item types reuse the existing `ScopeFilters` interface (`ItemFor<R> = ScopeFilters[R][number]`), so the resource is a checked `keyof ScopeFilters`, not a magic string.

Reads have three independent axes that mostly coincide but genuinely diverge: the **scope-token key** (into `scopeFor`), the **applyScope resource** (into `ScopeFilters`), and the **store slice**. They differ for `tcu` (slice `tcuActivities`), `trainees` (token `tcu`, slice `tcuTrainees` — the roster rides tcu visibility, ADR-0013), and `emailCampaigns` (token `bulkEmail`). These are stable facts about each entity, not per-call variation, so they live as data in one `satisfies`-typed `RESOURCE_READ: Record<keyof ScopeFilters, { scopeKey; slice }>`, where a wrong slice or an incoherent triple is a compile error. Every call site is therefore uniform — the subtle deviations that are easy to "helpfully fix" and break (changing `trainees` to `scopeFor['trainees']` empties the roster) sit in one reviewable table rather than scattered across files.

We rejected a `makeScopedApi` factory (mirroring `makeEntityMutation`): the api objects are heterogeneous — only some have `get`, only `courses` has `seatsRemaining` and a per-request `scopeOverride` — so a factory needs escape hatches that erode the depth it buys; bare helpers compose honestly with the variation. `courses.seatsRemaining` stays bespoke (a true aggregate, not a scoped list). The per-request `scopeOverride` (browse path, `useBrowseableCourse`) stays an optional param, orthogonal to the entity's fixed triple.

## Consequences

- The four-line pipeline exists once. A new scoped entity is a `RESOURCE_READ` row, a `Filters` interface, an `applyFilters`, and a one-line delegation — no pipeline to copy or mis-copy.
- `RESOURCE_READ` is `satisfies`-typed, so slice↔item↔token coherence is a compile-time guarantee; no runtime coherence sweep.
- Tests target the public api surface (`api.trainees.list()`, `api.programs.get()`), emphasizing the three deviant resources so the registry indirection is pinned; `scopedList`/`RESOURCE_READ` stay internal seams. `scope.test.ts` and `courses.test.ts` (the `applyScope` and `seatsRemaining` tests) are unchanged.
- This deepens ADR-0008's interpreter into one module without changing its contract: `scopeFor` still returns tokens, the api still interprets them. `scopedList` reads store state and passes it to `applyScope` per the state-parameter signature from the scope-seam change (Candidate 5), on which this sequences.
- Pre-existing and out of scope: `scopeOverride` living on the caller-facing `CourseFilters` is a small leak (a filter object carrying a scope concept). It is routed through the helper unchanged; removing it would touch the `useBrowseableCourse` call path and is left as a follow-up.
