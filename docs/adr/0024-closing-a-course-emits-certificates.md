# Closing a Course emits its Certificates; admin approval is removed

_Proposed (planning milestone "Certificate lifecycle rework + UI/UX overhaul"). Supersedes ADR-0019 and the approve half of ADR-0022 once the milestone lands._

Certificate approval is removed. A passing Grade (≥70) no longer mints a `pending` Certificate that waits for an admin (ADR-0019); instead a Teacher or admin **closes** the Course, and closing emits a downloadable Certificate for every enrolled Student with a passing Grade, all at once. `CourseStatus` gains `'closed'` (`draft | published | closed`) and the permission matrix gains a `close` action: `courses: { close: true }` for admin and `close: courseOwned` for a Teacher (ADR-0007), with the store rejecting a close unless the Course is currently `published` (ADR-0009). The `Certificate` type drops `status`, `approvedAt`, and `approvedBy`; `createdAt` becomes `issuedAt`, the close instant that dates the PDF. `isCertificateDownloadable` is now existence-based — a Certificate exists iff its Course was closed iff the PDF is available, so there is no longer an undownloadable Certificate to model.

We chose `'closed'` over `'ended'` deliberately: "ended" already means _the Term's end date has passed_ — a derived, time-based fact carried by the `courseOwnedAndEnded` grade predicate and the Teacher dashboard's "Ended Courses" card — which is distinct from this stored, deliberate ceremony a Teacher performs after reviewing Grades. We rejected emitting Certificates automatically at Term-end (the Teacher needs a review gate before credentials go out) and keeping approval as an optional confirmation (it recreates the bottleneck this removes; the close _is_ the confirmation).

## Consequences

- Supersedes ADR-0019: there is no pending state and no pending-first worklist. The Certificates page collapses to a role-scoped gallery whose only action is download; `approveCertificate`/`approveCertificates` and the `certificates: { approve }` matrix cells (admin and teacher) are deleted. Supersedes the approve half of ADR-0022: the in-course `CourseCertificatesSection` becomes list-only, surfacing the cohort's emitted Certificates after the Course closes.
- `closeCourse` is one store mutation through `withAudit` (ADR-0005) that both flips the status and emits the Certificates, attributed to the acting Teacher/admin (`currentUserId`); it declares every query key it invalidates (`['courses']`, the course detail key, `['certificates']`, `['students']`) so no scoped reader goes stale (the cache-invalidation contract that broke #87).
- The persisted shape changes (Certificate loses `status`; Course admits a new `closed` value), so `STATE_KEY` bumps `v9 → v10` and the `v9` key joins the dropped-legacy list — stale snapshots reseed, never migrate (ADR-0003). The seed marks completed cohorts `closed` and emits their Certificates as downloadable, dropping the old pending/approved/persona-pending logic.
- The admin dashboard's "Pendientes de aprobación" card and the WelcomeBanner pending count are removed, and no admin approval notification remains; the CONTEXT.md _Certificate_ definition is rewritten away from the pending/approved vocabulary.
