# Multi-query verdicts derive through a single resolveQueries seam

_Proposed (architecture review 2026-07-03)._

A page that derives a render decision from two or more React Query hooks must hold that decision until every one of them has loaded; reading a hook's default-`[]` window before it resolves computes a false verdict that flashes. This rule had no home: three pages hand-rolled it as OR-chains (`CoursesDetailPage`, `MarkSessionAttendancePage`) and one got it wrong — `TcuListPage` gated its roster table on the activities query's `isLoading` alone while each row read trainee names from a separate `useTcuTrainees()` query, so during the window where activities resolved first the table painted "undefined undefined" names. The render's data-dependency set was wider than its loading gate, and nothing made that gap visible.

The rule now lives in one pure function, `resolveQueries(results)`, which takes the results of existing scoped hooks and returns `{ isPending, data }`. It gates on each input's `isLoading` (not `isPending`), so a conditionally-disabled query — `enabled: false`, which sits at `status: 'pending'` forever — does not hang the gate, matching the hand-rolled behaviour it replaces. `data` is `undefined` until every input clears `isLoading`, then a positional tuple with non-undefined elements; there is no `[]` default, so a caller cannot read a half-loaded value — the `isPending` guard must be cleared before `data` narrows from `undefined` to the tuple. Under-specifying a render's data dependencies becomes a type error rather than a flash. A local verdict like the close-readiness checklist composes cleanly: `resolveQueries` owns the loading guard, the memo keeps its domain guards (course exists, `canClose`, Term ended) and the derivation.

The input is the structural `QueryLike<D> = { isLoading; data: D | undefined }`, not React Query's `UseQueryResult`, so the seam is decoupled from React Query's types and trivially faked in tests. We rejected a `useQueries({ combine })` migration (it wants the queries as an options array, abandoning the named scoped hooks that encapsulate keys/roles/api) and rejected a verdict-deriving variant `useDerivedVerdict(queries, fn)` (the only caller that derives — readiness — interleaves domain guards a generic helper would have to swallow; gate-only is the narrower thing that is actually common). No `isError` state: the fake-async list api has no throw path, so error handling would be a hypothetical second adapter; if real errors become reachable, extend the return then.

## Consequences

- `resolveQueries` is a pure function with no hooks or state, tested directly over three cases (all resolved, one loading, one disabled). It is not `use`-prefixed because it is not a hook.
- The transient-flash class is caught by first-paint regression tests: stub `api` `delay()` so the second query resolves slower than the first, then assert the _first_ painted frame synchronously — a waiting `findBy*` would miss the flash. Written for the two sites that regressed (`TcuListPage` blank name, `CoursesDetailPage` false Blocked); `MarkSessionAttendancePage`'s conversion is a pure refactor and needs no flash test.
- Gating on `isLoading` is load-bearing and must not be "simplified" to `isPending`: that would hang any page mixing always-on and conditionally-enabled queries on an infinite skeleton.
- A conditionally-disabled input contributes an `undefined` slot to the resolved tuple despite the non-undefined element type. This does not arise in practice — disabled queries (browse, seats) feed only the page gate, which reads `isPending`, never `data`; the `data`-reading callers pass always-enabled queries.
- New multi-query render decisions route through `resolveQueries` rather than a fresh OR-chain; a hand-rolled loading gate over 2+ hooks is the smell this replaces.
