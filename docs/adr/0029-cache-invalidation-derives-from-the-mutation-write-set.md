# Cache invalidation derives from the mutation's write-set

_Proposed (architecture review 2026-07-03). Supersedes ADR-0006._

Every store mutation routes through `withAudit(set, recipe)`, whose `next: Partial<StoreState>` is exactly the set of state slices the mutation wrote. That fact was previously discarded: each of the 26 mutation hooks in `src/hooks/api/*.ts` hand-declared an `invalidates: QueryKey[]` array, and those arrays drifted from what the store actually touched — `useDeleteEnrollment` omitted `['certificates']` (which `unenrollStudent` revokes) while spuriously listing `['courses']`; the same class broke #87's roster and shipped once as the ADR-0025 grade-score staleness. The same fact — a mutation's write-cascade — was asserted twice, as store fields and as query-key arrays, with no mechanical link.

Invalidation now **derives** from the write-set instead of being hand-maintained. `makeEntityMutation` snapshots store state before and after the (synchronous) store call and diffs top-level slices by reference; every mutation builds new arrays immutably, so reference inequality flags exactly the written slices. A single total `SLICE_TO_KEYS: Record<StoreSliceName, QueryKey[]>` maps each slice to the query-key prefixes it invalidates — identity by default, with the few documented edges where a slice feeds another entity's cache (`enrollments` → `['enrollments']` + `['courses']`, because seats and browse-eligibility derive from enrollments; `tcuActivities` → `['tcu']` + `['trainees']`, for trainee hour rollups) and the two name mismatches (`tcuActivities` → `['tcu']`, `tcuTrainees` → `['trainees']`). Because the map holds only list-key prefixes, React Query's default fuzzy matching covers every detail, scoped, and derived-child key (`['courses', id, role]`, `['courses', 'seats', id]`, …) with no per-id entry. The pure `writeSetInvalidations(before, after): QueryKey[]` is the deep module and the test surface; `makeEntityMutation` only wires it to `invalidateQueries`.

The `invalidates` config field is removed entirely — no per-hook escape hatch. A mutation that needs a key the diff cannot infer is, by definition, a cross-entity dependency, and it belongs in `SLICE_TO_KEYS` (the one audited place), not in a hook. `auditLog` stops being a special case: it is an ordinary slice that every `withAudit` mutation writes, so it is always in the diff and `['auditLog']` always invalidates. This deletes the ADR-0006 store subscription (`wireAuditInvalidation`), its wiring, and its test, and the explicit `AUDIT_LOG_KEY` line in `makeEntityMutation`; it also removes the dead `['dashboards']` key (no query is keyed there — `useDashboardStats` subscribes to the store directly).

We rejected keeping per-hook `invalidates` (the drift source itself), a static per-method write-set registry (relocated hand-maintained data that can still fall behind the real `next`), and a global `MutationCache.onSuccess` (fires regardless of what changed and couples invalidation to React Query plumbing). We chose deriving from the actual state diff because it is the only option where within-slice drift is impossible by construction.

## Consequences

- Within-slice invalidation drift is structurally impossible: a mutation invalidates the slices it actually wrote, not a hand-copied claim about them. The `useDeleteEnrollment`/`unenrollStudent` divergence cannot recur.
- The **CLAUDE.md** rule "a mutation must list every query key it `invalidates`" is reversed: mutations list nothing; invalidation is derived. Update that guidance when this lands.
- **ADR-0006 is superseded.** The `['auditLog']` cache is invalidated by the same uniform diff path as every other slice, not by a dedicated store subscription. This relies on all audited writes flowing through a `makeEntityMutation` hook — true today (no app code calls a store mutation directly); a mutation wired through a non-hook path would not refresh the audit page, which is already an anti-pattern here.
- `SLICE_TO_KEYS` is a total `Record` over the store's data-slice names, so adding a slice is a compile error until it is mapped. Cross-entity dependencies live in exactly one reviewable place.
- One residual risk class remains and is unaffected by the compiler: a query that derives from a slice not linked to its key prefix (a new cross-entity dependency the map does not yet encode). The diff cannot see it; the e2e suite is the primary guard, as it already was for #87.
- `makeEntityMutation`'s config shrinks to `{ toastKey, args? }` while its behaviour grows to include invalidation — a deeper interface. Reseed/reset is unchanged: it re-anchors `role` to `null`, and role lives in every query key, so the cache refreshes by key change independent of this mechanism (ADR-0014).
