# A Teacher views and approves Certificates for the Courses they own, and the Certificates page is a pending-first worklist

Certificate approval was admin-only: a Teacher — the person who actually taught the Course and saved the passing Grade that minted the Certificate — could neither see nor act on it. The permission matrix now grants `teacher` `certificates: { view: true, approve: courseOwned }` with a `certificates: 'ownCourses'` scope token (ADR-0007/0008), exactly mirroring how grades and attendance already scope to a Teacher's own Courses. `view: true` is unscoped on purpose: it opens the nav entry and route (which derive from the matrix, ADR-0010) without a context object, while the `'ownCourses'` data scope narrows the list and the `courseOwned` predicate gates `approve` per-Course. The store's `approveCertificate` resolves the Certificate's Course and passes it to `assertCan` (ADR-0009), so a Teacher approving outside their Courses throws at the real boundary — the same row-level enforcement pattern as `approveTcuActivity` (ADR-0017). Admin's unconditional `approve: true` is unchanged. We rejected giving Teachers a global approve cell (it would let any Teacher approve any Course's Certificate, breaking the One Sede / own-Course boundary) and a view-only Teacher (the Teacher is the natural approver; routing every approval through an admin recreates the bottleneck this removes).

The page itself is reshaped around the approver's actual job. For roles that can approve (admin, teacher) it becomes a **pending-first worklist**: a "Needs approval" segment (default) lists pending Certificates as dense, actionable rows with inline approve plus a bulk "Approve all", and an "Approved" segment keeps the browsable gallery whose action is download. The recipient view (a Student seeing their own Certificates, ADR-0012) is unchanged — pending still reads as "In review" with the download disabled. We rejected keeping the single mixed gallery (pending and approved interleaved buried the one action the approver came to perform) and a sortable table for every role (the Student recipient view is a keepsake, not a queue).

## Consequences

- Certificate visibility and approval are derived per-Course for Teachers: a new `'ownCourses'` branch on the certificates scope (ADR-0008) and an `approve` matrix predicate (ADR-0007), with the store hydrating the Certificate's Course into `can()` before mutating. No new permission resource is introduced; nav and route gating follow from the matrix automatically (ADR-0010).
- Bulk approval is a single store mutation over a list of Certificate ids — one audit entry through `withAudit` (ADR-0005), one toast, and it declares every query key it invalidates (the scoped certificates list and the dashboard's pending-approvals widget) so no scoped reader goes stale.
- No persisted shape changes: `Certificate` is unchanged, so `STATE_KEY` does not bump and snapshots need no reseed. Only the permission matrix, the scope interpreter, and the page change.
