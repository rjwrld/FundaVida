# Page-layer duplication collapses into a list-view discriminant and a student-progress card

_Proposed (architecture review 2026-07-03)._

Two page-layer duplications, extracted along different lines because they are duplications of different depth.

Eleven list pages hand-rolled the same four-state render — `isLoading ? Skeleton : count === 0 && !hasFilters ? Empty : count === 0 ? NoResults : Table` — but varied widely around it (different filters, DataTable vs hand-rolled tables, dialogs-or-not, some only three-state). The genuinely deep, repeated thing is not the surrounding structure but the branch _ordering_, which has a real mis-order bug surface. So the extraction is a pure `listViewState({ isLoading, count, hasFilters }) → 'loading' | 'empty' | 'noResults' | 'data'` plus a minimal `<ListView state skeleton empty? noResults content>` that renders only that branch — `empty` optional, falling back to `noResults`, which is what absorbs the three-state pages without a special case. Pages keep their `PageHeader`, filters, columns, and dialogs verbatim. We rejected a full `<ListPageShell>` component: the pages vary too much, so a component would need a slot per variation and become a shallow God-component leaking every page's specifics through its props. The discriminant is the deepest honest extraction here; the list-shell is deliberately the thinnest deepening in this review.

StudentsDetailPage and MeProfilePage were near-verbatim: the same `courseById/gradeByCourse/certByCourse/attendanceByCourse → enrollmentRows` aggregation and the same identity/enrollments/certificates/guardian sections, already drifting (the attendance cell rendered `<Progress>`+`<AnimatedNumber>` on one page and a plain percent on the other). This collapses into a pure `buildStudentProgress({ enrollments, courses, grades, attendance, certificates }) → StudentProgressRow[]` in `src/lib/` (the aggregation, tested directly) and a presentational `<StudentProgress student rows action />`, with the drift resolved toward the richer cell. The page owns its queries — `useStudent(id)` for admin/teacher, `useCurrentStudent()` for self — because that choice is the scope seam (ADR-0008/0012) and must not move into the shared card; we rejected a `mode: 'admin' | 'self'` prop for exactly that reason. The card receives resolved data, not query hooks. The concept the extraction names — a Student's per-Course roll-up of attendance, Grade, and Certificate — is added to `CONTEXT.md` as **Student Progress**, disambiguated from a TCU Trainee's hour progress.

The detail pages previously gated only on the student query and defaulted the other five to `[]`, risking the multi-query flash (a transient "notGraded" row before grades/attendance resolve). The progress section now gates on `resolveQueries([...])` (ADR-0030), so `buildStudentProgress` never runs on placeholder `[]`. This is why Candidate 4 sequences after Candidate 2.

## Consequences

- `listViewState` and `buildStudentProgress` are pure functions tested directly — the four-state ordering and the join/aggregate (including the skip-when-course-missing guard) are the two bug surfaces. `<ListView>` and `<StudentProgress>` are thin presentational shells covered by the surviving page tests; no dedicated render tests.
- Existing list-page and detail-page tests survive unchanged (they assert through rendered output), and the two detail-page tests still pin the scope-seam wiring — admin-by-id vs self.
- One first-paint asymmetric-delay test guards the new `resolveQueries` gate on the progress section (skeleton, not a false verdict, until all five resolve).
- The attendance-cell drift is resolved in the extraction's favour (the richer `<Progress>`+`<AnimatedNumber>` cell), so both pages render identically by construction.
- This is the widest-diff, lowest-per-site candidate; it sequences last. Splitting it into two PRs (list-view discriminant, student-progress card) keeps each reviewable.
